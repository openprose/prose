Runtime Backpressure

A managed agent runtime should explain why work can or cannot proceed before an operator clicks run. OpenProse keeps those reasons separate so capacity, providers, artifacts, approvals, and workspaces do not collapse into one vague failure state.

Operator intent

Create run component, inputs, provider, optional environment
Continue graph stored request context plus approved effect gates

Control-plane checks

capacity credits and concurrent run slots
provider readiness local-safe, Sprites, future hosted providers
artifact storage inline, local, or S3-compatible storage health

Decision surface

ready execution can start with durable run materialization
attention non-blocking issue: pending approvals, low Sprite pool, partial provider outage
blocked one concrete reason is attached to the disabled action

Backpressure checklist

Capacity Can this org afford and schedule another run? ready
Providers Which harnesses are available, default, or unavailable? ready
Storage Can artifacts be written and later inspected? ready
Approvals Are effect gates waiting for a human decision? attention
Sprite pool Are remote workspaces ready or replenishing? attention
Inputs Are required typed inputs and source pins present? blocked when missing

Provider rows answer

  • which provider is default
  • which harness, worker, or model will execute
  • whether a Sprite-backed environment is required
  • whether a git-backed source is required
  • the exact reason an unavailable provider is disabled

Why this matters

Backpressure turns a managed agent runtime from a text wall into an operating surface. Operators can see whether the next fix is adding inputs, approving an effect, waiting for capacity, allocating a workspace, fixing storage, or choosing a different provider.
