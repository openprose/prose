Hosted Approval Loop

Unsafe effects are now explicit enough that the platform can treat them as approval objects instead of burying them in runtime logs.

Overview Compiler and Runs Reactive Planning Packages and Registry Hosted Approval Loop

1. Ingest

compiled package + graph plan stored in hosted catalog/runtime tables

2. Detect gate

blocked_effect node `delivers`, `human_gate`, repo mutation, or other unsafe effect

3. Approval record

OpenProseApproval links graph, node, effect context, requester, and decision state

4. Refresh

explicit graph refresh operator re-plans after approval using the same stored request context

Why the approval object exists

  • it gives blocked work a first-class queue instead of a vague runtime error
  • it makes the decision reviewable in the operator surface
  • it preserves provenance between the planned node and the human/operator who unblocked it
